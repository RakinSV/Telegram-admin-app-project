## Зачем это

Владелец Telegram-канала обычно выбирает между двумя плохими вариантами:
руками пересказывать чужие посты каждый день, либо тупо форвардить с риском
жалоб на копипаст и потери аудитории. Этот проект автоматизирует весь путь
поста от источника до публикации — с рерайтом через LLM, проверкой на
дубли, модерацией перед выходом и статистикой, что реально заходит
аудитории. А отдельный **Guardian**-бот параллельно охраняет группу от
спама, ботов и рейдов, чтобы аудиторию, которую растит первый бот, не
затоптал второй волной накрутки.

Оба бота — production-ready: alembic-миграции с первого дня, 537 тестов,
CI с линтером/тайпчекером/security-сканером на каждый пуш, Docker-упаковка
для деплоя на VPS/Proxmox и единая веб-панель управления вместо ковыряния
в `.env` и БД руками.

Этот README — обзор и быстрый старт. **Пошаговая инструкция по деплою для
новичков с FAQ и решением частых проблем** — в **[Wiki](../../wiki)**;
если вы никогда раньше не запускали Docker-контейнер — начните оттуда, а не
отсюда. Полный технический контекст и архитектурные решения на уровне
реализации (для контрибьюторов/мейнтейнеров, не конечных пользователей) —
в [CLAUDE.md](CLAUDE.md), бэклог фич репост-бота — в
[FEATURES.md](FEATURES.md), план по фазам — в
[IMPLEMENTATION_PLAN.md](IMPLEMENTATION_PLAN.md), Guardian отдельно — в
[guardian/GUARDIAN.md](guardian/GUARDIAN.md) и
[guardian/GUARDIAN_FEATURES.md](guardian/GUARDIAN_FEATURES.md).

## Репост-бот — функционал

### Сбор и качество контента
- **Чтение без ограничений Bot API** — Telethon на юзер-сессии видит любые
  публичные/доступные каналы, не только те, где бот-администратор.
- **Фильтр по ключевым словам** — стоп-/обязательные слова глобально или на
  канал, применяется до сохранения в очередь.
- **Двухуровневая дедупликация** — быстрый хэш точных повторов + опциональный
  семантический дубль-чек через эмбеддинги (ловит перефразированные копии
  из разных каналов).
- **Стиль-профили рерайта** — новость / мнение / инструкция / юмор, разные
  промпт-шаблоны, привязка per-source или ИИ сам подбирает голос.
- **Добор источников** — Brave Search находит 2-3 релевантных ссылки
  (рус. + англ.) по теме поста, LLM отбирает лучшие, добавляется блок
  «📚 Источники» — рост доверия аудитории и защита от обвинений в фейках.
- **Сравнение версий события** — если русские и англоязычные источники
  расходятся в трактовке, пост честно помечается «⚠️ разные версии события».
- **Авто-обложки** — если у поста нет медиа, генерация через Unsplash API
  (быстро, бесплатно) или локальный ComfyUI (уникальная AI-генерация).

### Публикация и рост
- **Гибкая модерация** — ручное одобрение кнопками в личке / полный
  автопилот / дрип-постинг по временным слотам, любая комбинация.
- **Мультиканальная публикация** — свои цели на каждый источник
  (`Source.target_chat_ids`), не «всё во все группы».
- **Умное расписание** — анализ истории просмотров вычисляет пиковые часы
  аудитории и рекомендует (или автоматически применяет) сдвиг слотов
  публикации.
- **Авто-дайджест** — раз в неделю LLM сам собирает топ-постов в один
  сводный обзор.
- **Нативная реклама** — каждый N-й пост органично встраивает рекламный
  бриф в стиль канала, ротация по брифам.
- **Growth-трекер** — снимки числа подписчиков + счётчики постов по стилям,
  видно динамику роста канала.
- **Авто-реакция на негатив** — считает 👎/💩/😡-реакции, уведомляет
  владельца и опционально удаляет пост при превышении порога (с защитой от
  скоординированного бригадинга — лимит авто-удалений в час).

### Надёжность и масштаб
- **Антибан-механики** — джиттер между запросами, почасовые лимиты чтения —
  снижают риск ограничений юзер-сессии Telethon.
- **Ротация нескольких Telethon-сессий** — балансировка чтения источников
  между аккаунтами при росте числа каналов, отдельный rate-limiter на
  каждую сессию.
- **Статус-машина постов** — чёткие переходы `new → filtered_out |
  duplicate → rewriting → rewritten → pending_approval → approved |
  rejected → posted | failed`, каждый переход логируется.
- **Retry с экспоненциальной задержкой** на все сетевые вызовы (Telegram,
  LLM, поиск).
- **MTProto/SOCKS5-прокси** — Telethon (юзер-сессия) ходит через MTProto-
  прокси, Bot API (постинг/модерация) — через SOCKS5; см. [«Прокси»](#прокси)
  ниже — **и обязательно прочитайте про ограничение fake-TLS перед тем как
  настраивать.**

## Guardian — AI-модератор группы

Отдельный бот на `aiogram`, задача — не пропустить контент, а **защитить
живых людей в группе** от спама, ботов и токсиков, пока растёт аудитория.

### Верификация новых участников
- Мут сразу при входе → капча (математика / кнопка «я не робот» /
  тематический вопрос про канал) → снятие ограничений при верном ответе.
- Не ответил за N минут → автокик.
- Гонка «кто первый ответил» и путаница «кто на самом деле новый участник,
  а кто его пригласил» закрыты явной адресацией по `user_id`, а не через
  FSM-контекст aiogram (см. находку в истории коммитов — реальный
  security-баг, который нашли и закрыли).
- **Анализ профиля** — нет username/фото, подозрительное био, свежий
  аккаунт → усиленная капча для рискованных профилей (без авто-бана только
  за профиль — это было бы слишком агрессивно).

### Спам-фильтр — три режима на выбор
| Режим | Как работает | Когда использовать |
|---|---|---|
| `keywords` | Стоп-слова с нормализацией против обхода (омоглифы, zero-width символы, разделители) | Ноль стоимости, для маленьких/тихих групп |
| `ai` | Каждое сообщение — классификатор через OpenAI-совместимое API, JSON-ответ с confidence | Максимальное качество, но токены на каждое сообщение |
| `hybrid` | Эвристики (цены, «пиши в лс», zero-width, форварды, новый аккаунт) отбирают ~20% подозрительных → только они идут в AI | **Рекомендуемый режим** — качество AI по цене keywords |

AI-классификатор **fail-open**: любая ошибка, таймаут или невалидный JSON
трактуется как «пропустить», а не «удалить» — при неуверенности бот никогда
не режет легитимное сообщение.

### Модерация и защита
- **Whitelist-фильтр ссылок**, включая скрытые за текстом (`text_link`).
- **Антифлуд** — лимит сообщений за окно + детект дублей подряд.
- **Система варнов** с эскалацией по наивысшему достигнутому порогу (мут →
  кик → бан), TTL-автосброс варнов по расписанию.
- **Антирейд** — всплеск вступлений замораживает права всего чата
  (с сохранением прежних прав, не дефолтных), кнопки ручной/авто
  разморозки при затишье.
- **Тихие часы** — расписание авто-переключения между строгим (варн за
  любое нарушение) и мягким (только логирование ссылок) режимом.
- **Автодоверие** — участники без нарушений N дней автоматически обходят
  фильтры, снижая AI-нагрузку на активную, уже проверенную аудиторию.
- **Лог-канал** — каждое действие модерации, ручное и автоматическое,
  пишется в приватный канал с inline-кнопками для быстрой реакции админа.
- **SOCKS5-прокси для Bot API** — см. [«Прокси»](#прокси) ниже.
- **21 команда администратора** — `/warn /mute /unmute /ban /unban /kick
  /check /addword /delword /listwords /trust /untrust /addomain /deldomain
  /listdomains /setmode /setcaptcha /setwarn /setmutime /mode /stats
  /growth` — весь конфиг меняется на лету, без перезапуска бота.

## Веб-админка

Единая FastAPI-панель на оба бота, встроена в тот же процесс, что и
`main.py` (не отдельный сервис) — настройки применяются живо, логи
стримятся в браузер без межпроцессной синхронизации.

- **`/setup`** — визард первого запуска: пароль администратора + Telegram
  вход прямо в браузере (телефон → код → 2FA), без ковыряния в терминале.
- **`/`** дашборд — воронка постов по статусам, токены рерайта за сегодня,
  доля ошибок за 24ч, статус компонентов (listener/бот/планировщик).
- **`/sources` `/targets` `/moderation` `/ads` `/stats`** — полный CRUD
  зеркалит CLI и Telegram-бота на одной и той же бизнес-логике (не две
  версии кода, которые могут разойтись).
- **`/settings`** — ~25 настроек по фиче-группам, применяются сразу live;
  поля с пометкой `resync` автоматически пересобирают джобы планировщика.
- **`/secrets`** — write-only форма: значение шифруется (`Fernet`) и
  никогда не возвращается в браузер, только маска `••••a1b2`.
- **`/components`** — живой рестарт Telethon-листенера/бота без
  перезапуска процесса после смены токена/сессии.
- **`/guardian*`** — управление Guardian из той же панели: режим
  спам-фильтра, пороги варнов, стоп-слова, whitelist доменов, доверенные
  пользователи — без единой Telegram-команды.
- **`/audit`** — журнал всех действий из админки (кто, что, когда).
- **`/logs`** — живые логи процесса через Server-Sent Events, без
  WebSocket-инфраструктуры.

Доступ — только localhost/VPN по дизайну (без обязательного TLS), сессии
с idle- и абсолютным таймаутом, rate-limit на `/login`, Argon2id для
пароля.

## Прокси

Telethon (юзер-сессия) говорит с Telegram напрямую по **MTProto** — обычный
SOCKS5/HTTP-прокси тут не встанет, нужен именно **MTProto-прокси**. Bot API
(постинг/модерация обоих ботов), наоборот, ходит по HTTPS — там MTProto
бесполезен, нужен **SOCKS5**. Многие провайдеры под Telegram-автоматизацию
продают оба эндпоинта на одном сервере.

| Что проксируется | Тип | Где настраивается |
|---|---|---|
| Telethon (основная сессия + вся ротация F26) | MTProto | `/settings` (host/port) + `/secrets` (secret) — один общий прокси на все Telethon-клиенты |
| Bot API репост-бота | SOCKS5 | `/secrets` (`Bot API Proxy URL`, целиком, включая логин:пароль) |
| Bot API Guardian | SOCKS5 | **только `.env`** (`GUARDIAN_BOT_API_PROXY_URL`) — Guardian, в отличие от репост-бота, не умеет живой рестарт компонентов, `Bot()` строится один раз при старте процесса, как и `GUARDIAN_BOT_TOKEN`; смена требует `docker compose restart guardian` |

Пусто = прямое соединение без прокси (поведение по умолчанию, ничего
настраивать не обязательно).

> ⚠️ **Прочитайте это перед настройкой MTProto-прокси.** Встроенная
> поддержка MTProto-прокси в Telethon **не умеет** режим fake-TLS (секреты
> с префиксом `ee`) — это известное ограничение самой библиотеки, а не то,
> что можно починить в этом проекте. Большинство современных публичных
> MTProto-прокси по умолчанию работают именно в fake-TLS. Если ваш секрет
> начинается с `ee` — Telethon либо зависнет, либо упадёт с ошибкой
> рассинхронизации шифрования, что бы вы тут ни настроили. Нужен классический
> секрет (обычный hex, либо с префиксом `dd`) с того же прокси, либо другой
> прокси-сервер. **Подробности, реальные тексты ошибок и алгоритм принятия
> решения — в [Wiki, раздел «Прокси»](../../wiki/Proxy-Guide-RU).**

## Стек

| Слой | Технология | Почему |
|---|---|---|
| Чтение каналов | **Telethon** (юзер-сессия) | Bot API не даёт читать чужие каналы без прав администратора |
| Публикация/модерация | **python-telegram-bot** (репост) · **aiogram 3.x** (Guardian) | aiogram — FSM из коробки для многошаговой капчи |
| Рерайт/классификация | **OpenAI SDK**, любой совместимый `base_url` | GPT-4o-mini, Claude через прокси, локальная llama.cpp/Ollama — без смены кода |
| БД | **SQLite + SQLAlchemy + Alembic** | Простота, один файл, лёгкий бэкап; путь на Postgres открыт с первого дня |
| Планировщик | **APScheduler** | Cron-джобы внутри процесса, без Celery/Redis для одного оператора |
| Веб-админка | **FastAPI + Jinja2 + Starlette Sessions** | Встроена в общий event loop, live-reload настроек без пересборки фронта |
| Секреты | **Fernet (cryptography)** + Argon2id | Симметричное шифрование at rest, write-only UI |
| CI | **GitHub Actions**: ruff · mypy · pytest · bandit · pip-audit | Каждый пуш проверяется линтером, тайпчекером и двумя security-сканерами |
| Деплой | **Docker + docker-compose** | Два сервиса на общем образе, разные entrypoint, изолированные volume |

## Docker-деплой

Это рекомендуемый способ запуска проекта. Если вы никогда не работали с
Docker — читайте дальше, но для максимально подробного, беспроблемного
пошагового гайда со скриншотами типичных ошибок используйте
**[Wiki: Installation-RU](../../wiki/Installation-RU)** вместо этой
сокращённой версии.

### 1. Установить Docker

**Debian/Ubuntu:**
```bash
curl -fsSL https://get.docker.com | sh
sudo usermod -aG docker $USER
# Выйдите и зайдите заново (или выполните `newgrp docker`) — членство в
# группе не применится к текущей сессии терминала, пока вы этого не сделаете.
```

Проверка:
```bash
docker compose version
```
Если ругается что-то вроде `unknown command "compose"` — значит стоит
старый standalone `docker-compose` (или вообще ничего), а не плагин
Compose V2, на который рассчитаны команды в этом проекте. Установить:
```bash
sudo apt update && sudo apt install docker-compose-plugin
```

**Windows/macOS:** поставьте [Docker Desktop](https://www.docker.com/products/docker-desktop/) — Compose V2 уже встроен.

### 2. Склонировать репозиторий

```bash
git clone https://github.com/RakinSV/Telegram-admin-app-project.git
cd Telegram-admin-app-project
```

### 3. Создать `.env` — **до** первого `docker compose up`

```bash
cp .env.example .env
```

Этот шаг не опционален, и порядок важен: если `.env` ещё не существует,
bind mount Docker'а (`./.env:/app/.env` в `docker-compose.yml`) создаст на
его месте **директорию**, а не примонтирует файл — контейнер упадёт с
непонятной ошибкой. Заполнять руками ничего не обязательно — оставьте все
секреты пустыми, зададите их позже через веб-визард `/setup`. Единственное,
что стоит сделать сейчас — открыть `.env` и пробежаться по комментариям,
чтобы понимать, что там вообще есть.

### 4. Запустить

```bash
docker compose up -d --build
```

Первый билд займёт пару минут (тянутся зависимости). Два сервиса на одном
образе — `tg_repost` (веб-админка + репост-пайплайн, публикуется на
`127.0.0.1:8000`) и `guardian` (модерация группы) — с разными entrypoint,
независимым рестартом (`unless-stopped`), общим `.env` и раздельными volume
для БД/логов/медиа.

Проверить, что всё поднялось:
```bash
docker compose ps
docker compose logs -f tg_repost
```

### 5. Открыть админку

Порт публикуется как `127.0.0.1:8000` — **только loopback хоста**, это
осознанное решение, тот же периметр localhost/VPN-only, что и без Docker.
Если вы на той же машине — открывайте `http://127.0.0.1:8000/setup` в
браузере. Если деплоите на **удалённый сервер** — есть два варианта, оба
подробно разобраны с точными командами в
**[Wiki](../../wiki/Installation-RU#шаг-6--открыть-админку)**:
- SSH-туннель с ноутбука (`ssh -L 8000:127.0.0.1:8000 user@server`), либо
- `docker-compose.override.yml`, публикующий порт на VPN/LAN-интерфейс
  вместо `127.0.0.1` (никогда не публикуйте эту панель напрямую на
  публичный IP без reverse-proxy и TLS перед ней).

При первом запуске в логи печатается одноразовый токен для `/setup` —
достать его:
```bash
docker compose logs tg_repost | grep -i token
```

Дальше: создаёте пароль администратора, затем кнопкой **«Войти через
Telegram»** на странице `/secrets` подключаете Telethon-аккаунт (телефон →
код → 2FA, прямо в браузере, без терминала).

### 6. Обновление

```bash
git pull
docker compose up -d --build
```
`.env` и обе базы данных (в `./data/`) не трогаются — они живут вне
образа, в примонтированных volume.

### Частые проблемы

| Симптом | Причина | Решение |
|---|---|---|
| `unknown command "compose"` | Не установлен плагин Compose V2 | `sudo apt install docker-compose-plugin` |
| `permission denied ... docker.sock` | Членство в группе после `usermod -aG docker` ещё не применилось | Перелогиньтесь, или `newgrp docker` |
| `.env` создался как директория | `docker compose up` запустили до того, как `.env` появился | `docker compose down`, `rmdir .env` (она пустая), `cp .env.example .env`, повторить |
| `/guardian*` падает 500 "no such table" | Рассинхронизация `GUARDIAN_DATABASE_URL` между сервисами, либо устаревший образ до фикса | `docker compose up -d --build`, чтобы пересобрать оба сервиса из текущего `docker-compose.yml` |
| Вход в Telethon зависает или падает с непонятной ошибкой | MTProto-прокси в режиме fake-TLS — не поддерживается, см. [«Прокси»](#прокси) выше | Читайте [Wiki: Proxy-Guide-RU](../../wiki/Proxy-Guide-RU) |

Больше сценариев с реальными текстами ошибок — в
**[Wiki: FAQ-RU](../../wiki/FAQ-RU)**.

## Бэкап

```bash
python -m tg_repost.tools.backup            # хранит последние 14 бэкапов
0 3 * * * cd /path/to/repo && python -m tg_repost.tools.backup   # cron
```
Архивирует `.env` + обе SQLite БД + `logs/` в один zip — без `.env`
зашифрованные секреты в БД невосстановимы, поэтому бэкапятся вместе.
Права файла архива ограничены `0600`; не синкать `backups/` в облако без
отдельного шифрования (`age`/GPG).
