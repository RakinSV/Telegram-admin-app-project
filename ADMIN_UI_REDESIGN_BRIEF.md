# ТЗ на редизайн веб-админки tg_repost + Guardian

> Цель этого документа — дать дизайн-проходу (другой Claude-сессии/UI-дизайнеру)
> полную и точную карту того, что СЕЙЧАС есть в админке: какие страницы,
> что на них, какие действия доступны, какая дизайн-система уже используется,
> и какие конкретные UX-проблемы накопились за несколько итераций разработки.
> Составлено на основе полного чтения всех 26 Jinja2-шаблонов и всех роутов
> (`app.py`, `crud_routes.py`, `guardian_routes.py`) — не на предположениях.

## Что нужно от редизайна

**Сделать лаконичнее и удобнее**, не ломая функциональность:
- Все текущие роуты, формы, поля, названия полей `name=` — **не менять** без
  явной необходимости (бэкенд их ожидает один в один). Редизайн — это
  вёрстка/раскладка/визуальная иерархия/навигация внутри страницы, не
  переписывание бизнес-логики.
- Если для лаконичности нужно что-то физически убрать со страницы (например,
  свернуть 16 групп настроек в аккордеон/табы) — это ОК, это именно та задача,
  которая нужна. Просто это не должно менять `action=`/`method=`/`name=` полей.
- Один сквозной визуальный язык вместо смеси паттернов, накопившихся за много
  сессий разработки (см. раздел «Сквозные проблемы» ниже — это готовый чек-лист
  несостыковок для исправления).

## Текущая дизайн-система (`tg_repost/webui/static/style.css`)

- Единственная тёмная тема, без light-режима, без media-queries вообще —
  **не адаптивно**: сайдбар фиксированной ширины 220px, контент максимум 980px.
- Цвета (CSS-переменные): `--bg #0f1115`, `--panel #181b22`, `--border #2a2f3a`,
  `--text #e4e6ea`, `--muted #8b909c`, `--accent #5b8def` (синий),
  `--ok #3ec97a` (зелёный), `--warn #e0a82e` (янтарный), `--err #e0524f` (красный).
- Раскладка: `.layout` = flex-ряд из `nav.sidebar` + `main`. Стат-плитки —
  `.grid { grid-template-columns: repeat(auto-fit, minmax(160px,1fr)) }`.
- Типографика: системный шрифтовой стек, база 14px. Только `h1` (20px) и
  `h2` (15px, uppercase, приглушённый) — нет полноценной шкалы заголовков/
  подписей.
- Базовые компоненты: `.card` (единственный «блок» — используется буквально
  на каждой секции каждой страницы), `.badge` (4 варианта: ok/warn/err/muted),
  `.stat .num`/`.stat .label` (крупное число + подпись), голые `table/th/td`
  без зебры/сортировки/фильтров, `.check-list`/`.check-item` (используется
  ТОЛЬКО на `source_detail.html`), `.field-row` (label+input, **захардкожена
  ширина label 200px** — ломается на длинных подписях), `.inline-form`
  (однострочная форма добавления).
- Кнопки: primary (заливка accent), `.secondary` (серая заливка), `.danger`
  (красный контур), модификатор `.small`. Нет стиля для disabled, нет
  icon-button.
- Инпуты стилизуются только через селекторы-потомки (`.card input[type=text]`
  и т.п.) — инпут вне `.card`/`.field-row` откатывается на браузерный дефолт.
- `.auth-page` — узкая центрированная колонка (max-width 360px) для
  login/setup.
- **Мёртвый CSS**: `nav.sidebar a.active` определён, но НИГДЕ не применяется —
  сайдбар никогда не подсвечивает текущую страницу.
- Нет системы иконок — всё через сырые эмодзи (✅❌✏️🔗🖼⚠️📈📊🏆📚),
  выбраны несистемно, по вкусу конкретной сессии разработки.

## Полная карта страниц

### Публичные / авторизационные страницы

**Login** — `GET/POST /login`, шаблон `login.html` (без сайдбара).
Пароль-only форма. Rate-limit по IP (5 попыток/30 сек → 429 с текстом).
Ничего лишнего — ни «запомнить меня», ни показа пароля.

**Первый запуск** — `GET/POST /setup`, шаблон `setup.html`.
Один большой multi-purpose форм: пароль администратора (мин. 8 симв.) +
опциональный блок «Telegram» (`tg_api_id/hash`, `tg_bot_token`,
`tg_owner_user_id` с пояснениями где брать) + опциональный блок «Рерайт»
(`openai_api_key`). Это САМАЯ плотная страница публичной части — три разных
по смыслу блока стоят друг под другом только с `<h2>`-разделителями, без
карточек-контейнеров (в отличие от `/settings`, где каждая группа — своя
`.card`). Защищена одноразовым `?token=`.

**Setup locked** — фоллбэк на 403, если `/setup*` открыт без валидного токена.
Статический текст, ничего интерактивного.

**Вход через Telegram (Telethon-визард)** — `GET/POST /setup/telethon[...]`
и `GET/POST /components/telethon[...]`, ОДИН шаблон `telethon_login.html` на
оба контекста (до бутстрапа — без сайдбара, после — с сайдбаром). 3 шага:
телефон → код → опционально пароль 2FA → готово. Есть кнопка отмены
(закрывает живой Telethon-клиент, если бросить визард на середине).

### Главный дашборд

**`GET /`** — шаблон `dashboard.html`. Статус компонентов (badge-плитки),
воронка постов по статусам, токены рерайта за сегодня, error rate за 24ч,
таблица «Последние посты» (ID, Вид, Статус, Создан, Текст обрезан до 80 симв).
Read-only, единственный интерактив — предупреждение со ссылкой на `/secrets`
(⚠️ **устаревшая ссылка** — этот роут теперь редиректит на `/settings`,
не обновили после слияния настроек и секретов).

### Контент-пайплайн

**Источники** — `/sources`. Таблица: ID, Активен, Username (ссылка на
детали), Стиль, Добор (вкл/выкл/по умолчанию), Цели (число или «все»).
Форма добавления одним полем (`channel`). «Деактивировать» — без подтверждения.
Обрезается на 500 строк, ни поиска, ни пагинации.

**Детали источника** — `/sources/{id}`. `<select>` стиля, `<select>` режима
добора, чек-лист целей публикации (`.check-list` — используется ТОЛЬКО тут,
нигде больше в приложении), «осиротевшие» chat_id (были выбраны, но цель
удалена) показываются с предупреждающим badge, чтобы не потерять их молча.
Одна форма сохраняет все три поля разом.

**Цели публикации** — `/targets`. Таблица: ID, Активна, chat_id, Title.
Форма добавления (chat_id + опционально title). Единственная страница с
ДВУСТОРОННИМ toggle («Активировать»/«Деактивировать») вместо одностороннего
деактивирования.

**Модерация (очередь)** — `/moderation`. Таблица: ID, Тип, Статус
(badge всегда `.warn`, не завязан на реальное значение — тут показываются
только «на модерации» посты), Текст (80 симв.), Создан, ссылка «Открыть».

**Модерация (детали поста)** — `/moderation/{id}`. Полный редактируемый
текст (`<textarea rows=14>`), ссылка на источник, текстовая пометка «🖼 Есть
медиа» — **без реального превью картинки/видео**. Три отдельные формы:
«Сохранить текст», «✅ Одобрить», «❌ Отклонить» (approve/reject могут упасть
с ошибкой, если бот модерации не запущен — показывается инлайн).

**Реклама (нативные брифы)** — `/ads`. Таблица: ID, Активен, Использован,
Лимит (или ∞), Текст (80 симв). Форма добавления (текст + опционально
max_uses). «Деактивировать» — только для активных.

**Telethon-сессии (доп. аккаунты)** — `/telethon-sessions`. Таблица: ID,
Активна, Метка, Маска. Форма добавления: `label` + `session_string` как
password-инпут — сессию нужно получить ОТДЕЛЬНО через CLI-утилиту, никакого
визарда тут нет (асимметрия с основным аккаунтом, у которого полноценный
пошаговый вход через браузер).

### Статистика

**Обзор** — `/stats`. 4 стат-плитки (опубликовано / учтено / просмотров
всего / среднее), карточка «топ-пост» или пустое состояние, две текстовые
ссылки на best-times/growth (просто `→`, не оформлены как под-навигация/табы,
хотя концептуально это один раздел «Статистика»).

**Лучшее время** — `/stats/best-times` + `POST .../apply`. Порог «недостаточно
данных» (показывает сколько есть / сколько нужно), иначе — badge-пилюли
рекомендованных часов + кнопка «Применить сейчас» (пишет в настройки,
триггерит ресинк планировщика, зелёная карточка подтверждения).

**Рост подписчиков** — `/stats/growth`. Порог «недостаточно данных», иначе —
3 плитки (было/стало/дельта со знаком), опциональная таблица «по стилям»
(Стиль, Постов), сноска о том, что это счётчики, не статистическая корреляция.

### Настройки и секреты — САМАЯ ПЛОТНАЯ страница

**`GET /settings`, `POST /settings/{group_key}`,
`POST /secrets/{key}[/clear|/reveal]`** — шаблон `settings.html`.

Одна длинная страница, рендерит **16 групп настроек** последовательно, каждая
— свой `<h2>` + описание + `.card`-форма, плюс инлайн-подблок «Секреты группы»
у тех групп, к которым привязаны секреты:

1. **Telegram (идентичность)** — API ID, Owner user ID + секреты (API Hash,
   Bot Token, Session String)
2. **Прокси** — MTProto host/port + секреты (MTProto secret, Telethon SOCKS5,
   Bot API SOCKS5)
3. **Рерайт — F06** — Base URL, модель + секрет (OpenAI API Key)
4. **Фильтрация по словам — F03** — стоп-/обязательные слова
5. **Пайплайн** — интервал тика (resync), авто-постинг, уровень логов
6. **Антибан — F17** — мин/макс задержка, лимит чтений в час
7. **Расписание публикации — F11** — вкл (resync), слоты (resync),
   постов за слот
8. **Семантический дубль-чек — F13** — вкл, модель эмбеддингов, порог, окно
9. **Статистика — F14** — вкл (resync), интервал (resync), окно
10. **Реакция на негатив — F25** — порог, авто-удаление, потолок в час
11. **Стиль-профили — F15** — профиль по умолчанию
12. **Добор источников — F16** — вкл, Brave URL, лимиты + секрет (Brave key)
13. **Авто-обложки — F18** — вкл, стратегия `<select>`, Unsplash/ComfyUI поля
    + секрет (Unsplash key)
14. **Умное расписание — F19** — пороги, авто-применение (resync)
15. **Авто-дайджест — F20** — вкл (resync), день/час/минута (resync), топ-N, окно
16. **Нативная реклама — F21** — каждый N-й пост
17. **Growth-трекер — F22** — вкл (resync), интервал снимков (resync), пороги

Поля рендерятся по типу: `<select>` если заданы `choices`, чекбокс для bool,
текст через запятую для csv_list, `<input number>` для int/float. Поля с
`needs_resync` помечены `.badge.warn` "resync".

Секреты в подблоке: label, статус-badge (маска или «не задан»), источник
(веб-админка/.env/—), описание/подсказка, форма «новое значение + Сохранить»,
кнопка **«Показать»** (требует повторного ввода пароля администратора,
показывает значение один раз, не кэшируется), кнопка **«Очистить»** (только
если источник — БД) — **единственная кнопка во всём приложении с JS
`confirm()`-диалогом**.

**Это самая плотная/захламлённая страница всего приложения** — 16 карточек
подряд на один скролл, без под-навигации/табов/якорей/сворачивания —
чтобы долистать до «Growth-трекера» внизу, нужно проскроллить мимо 15
несвязанных групп.

### Компоненты (рантайм-статус)

**`GET /components`** + `POST /components/{start|listener/restart|bot/restart|
scheduler/resync}`. 3 стат-карточки (Listener/Бот модерации/Планировщик) со
своими кнопками рестарта, предупреждение если ничего не запущено + кнопка
«Запустить сейчас», ссылка на визард смены Telegram-аккаунта.

⚠️ **Коллизия имени**: страница называется «Компоненты» и имеет в виду
бэкенд-процессы (listener/bot/scheduler), НЕ UI-компоненты — если в редизайне
появится UI-кит с документацией, не путать термин.

### Журнал изменений

**`GET /audit?page=N`**. Таблица: Время, Действие (сырые snake_case-ключи
типа `source_add`, `secret_reveal`, `guardian_setting_set` — не
человекочитаемые), Объект, Детали. **Единственная страница с пагинацией**
во всём приложении.

### Логи

**`GET /logs`, `GET /logs/stream` (SSE)**. Пре-рендер последних строк +
живой стрим через `EventSource` (индикатор соединения: подключение…/живо/
переподключение…). **Единственная страница с реальным JS-интерактивом** —
всё остальное приложение живёт на классических POST + 303-редиректах.

### Guardian (бот-модератор группы) — общий логин с tg_repost

**Дашборд** — `/guardian`. Предупреждение если не настроен (токен/group_id —
редактируются только в `.env`, не отсюда). Плитки текущего конфига (режим
спам-фильтра, тип капчи, пороги мут/кик/бан). Счётчики (стоп-слова, домены,
доверенные — с переходом на свои страницы; участники/забаненные — **только
отображение, нет отдельной страницы управления**). Таблица последних действий
модерации: Когда, Действие, Пользователь (буквально строка `id{user_id}`, не
имя), Причина, Кто.

**Настройки Guardian** — `/guardian/settings`. Структурно похож на общий
`/settings`, но проще — без секретов, без `needs_resync` (Guardian-настройки
документированы как «всегда читаются свежими»). **9 групп**: identity,
spam_filter, captcha, warns, flood, raid, trust, profile, quiet_hours.
Тоже упоминает устаревшую `/secrets`-ссылку (токен бота/OpenAI-ключ
редактируются на странице репост-бота).

**Стоп-слова** — `/guardian/stopwords`. Таблица (Слово + действие), форма
добавления одним полем, «Удалить» без подтверждения.

**Whitelist доменов** — `/guardian/domains`. Идентичный стоп-словам паттерн.

**Доверенные пользователи** — `/guardian/trusted`. Таблица: user_id,
Добавлен, Кем, Причина, действие. Хрупкий инлайн-хак: `margin-left:288px`
чтобы выровнять текст подсказки под лейблом — завязан на захардкоженную
ширину `.field-row` label (200px), сломается при любом изменении пропорций.

## Сквозные проблемы (готовый чек-лист для редизайна)

1. **Повторяющийся, но не абстрагированный паттерн списка.** `sources.html`,
   `targets.html`, `ads.html`, `telethon_sessions.html`,
   `guardian_stopwords.html`, `guardian_domains.html`, `guardian_trusted.html`
   — все реализуют один и тот же рецепт (h1 → описание → карточка с формой
   добавления → карточка с таблицей → кнопка действия в строке) с нуля в
   каждом шаблоне — очевидный кандидат на единый переиспользуемый
   «список»-паттерн.
2. **Разнобой глаголов для деструктивных действий**: «Деактивировать»
   (источники/реклама/сессии), «Активировать»/«Деактивировать» (цели),
   «Удалить» (стоп-слова/домены), «Убрать» (доверенные), «Очистить»
   (секреты) — все означают примерно одно и то же.
3. **Разнобой в подтверждении деструктивных действий**: только «Очистить»
   секрет имеет JS `confirm()`; все остальные удаления/деактивации срабатывают
   мгновенно по клику без подтверждения.
4. **Устаревшие ссылки на `/secrets`** — в `dashboard.html`, `components.html`,
   `guardian_settings.html` — видимый шов от слияния настроек и секретов.
5. **Мёртвый CSS активного пункта навигации** — сайдбар никогда не подсвечивает
   текущую страницу.
6. **`/settings` и `/guardian/settings` — самые захламлённые страницы** —
   16 и 9 карточек подряд на один скролл без под-навигации/табов/якорей.
7. **Нигде нет пагинации/поиска/сортировки/массовых действий, кроме `/audit`**
   — списки просто обрезаются на 500 строках с текстовым предупреждением.
8. **Только одна страница с реальным интерактивом** (`/logs`, SSE) — всё
   остальное живёт на full-page POST + редиректах, включая переключение
   одного тумблера.
9. **Смешанные визуальные языки состояния**: badge-система — систематическая,
   но страницы вперемешку используют и сырые эмодзи как индикаторы статуса
   (иногда на той же странице, что и badge — например, дашборд).
10. **Риск коллизии имени**: «Компоненты» = бэкенд-процессы, не UI-компоненты.
11. **Модерация «слепа» к медиа** — только текстовая пометка «есть медиа»,
    нет превью картинки/видео нигде в флоу ревью.
12. **Асимметричный UX Telethon**: основной аккаунт — полный гид (телефон/
    код/2FA), дополнительные — только через CLI + ручная вставка строки сессии.
13. **Хрупкие инлайн-хаки вёрстки** — например, `margin-left:288px` в
    `guardian_trusted.html`, завязанный на захардкоженную ширину лейбла.

## Что НЕ входит в задачу

- Не менять роуты, `name=` полей форм, поведение бэкенда, статус-коды.
- Не трогать секьюрити-механизмы (rate-limit, reveal-флоу секретов,
  одноразовый setup-токен) — только их визуальное представление.
- Не обязательно чинить все 13 сквозных проблем разом одним PR — можно
  приоритезировать, но каждая должна быть в итоге закрыта осознанно (либо
  исправлена, либо явно решено оставить как есть с объяснением).
