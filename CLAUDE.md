# Telegram Content Repost System — контекст проекта

> Этот файл — точка входа для Claude Code. Прочитай его первым в начале каждой сессии.
> Остальные файлы: `FEATURES.md` (полный бэклог фич), `IMPLEMENTATION_PLAN.md` (план по фазам).

## Что это за проект

Система парсинга постов из выбранных Telegram-каналов, рерайта их через
OpenAI-совместимое API (для уникальности контента) и автоматической/ручной
публикации в собственные Telegram-группы/каналы пользователя.

Автор и единственный разработчик — Серёга, 17+ лет в автоматизации (1С, ERP,
интеграции), архитектор-первый разработчик, работает самостоятельно. Стек и
архитектурные решения должны соответствовать его опыту: он силён в Python,
архитектуре, БД, серверном администрировании (Linux, Windows Server,
MikroTik), и уже работал с Claude API / OpenAI API / RAG до этого.

## Ключевое решение по стеку

- **Язык**: Python 3.11+
- **Парсинг Telegram**: Telethon (юзер-сессия, а не Bot API — это позволяет
  читать любые публичные/доступные каналы без ограничений Bot API, которые
  не дают боту читать чужие каналы)
- **Постинг**: Telegram Bot API (через `python-telegram-bot` или `aiogram`) —
  для публикации в свои группы это безопаснее и проще, чем постить от юзера
- **БД**: SQLite на старте (простота, один файл, легко бэкапить). Возможен
  переход на Postgres при росте нагрузки/масштаба (см. план).
- **Рерайт**: любой OpenAI-совместимый API (`base_url` + `api_key` в конфиге) —
  GPT-4o-mini, Claude API через совместимый прокси, локальная модель через
  llama.cpp/Ollama. Один клиент, провайдер меняется в конфиге, не в коде.
- **Планировщик**: APScheduler (cron-like задачи внутри Python-процесса, без
  отдельного celery/redis на старте — избыточно для одного пользователя)
- **Запуск**: локально на своей машине или VPS, без Docker на первом этапе
  (можно добавить позже для портируемости)

## Архитектура (высокий уровень)

```
Источники (N Telegram-каналов)
        ↓
Telethon listener — слушает новые посты, фильтрует по ключевым словам,
                     дедуплицирует (хэш + семантическая схожесть через эмбеддинги)
        ↓
SQLite очередь (таблица posts, статус-машина: new → rewriting → rewritten →
                approved/rejected → posted)
        ↓
Рерайтер — OpenAI-совместимое API, промпт-шаблоны по типу контента
        ↓
[опционально] Поиск доп. источников — Brave Search / SerpAPI по теме поста,
              добавление блока "Источники" со ссылками рус./англ.
        ↓
Модерация — две ветки:
  1. Ручное одобрение: Telegram-бот шлёт пост в личку с кнопками
     ✅ Одобрить / ❌ Отклонить / ✏️ Редактировать
  2. Авто-постинг по расписанию (APScheduler), переключается флагом
        ↓
Постер — Telegram Bot API, публикация в одну или несколько целевых групп
        ↓
Статистика и логи — просмотры, реакции, источники, что зашло лучше всего
```

## Идеи, прозвучавшие в обсуждении (полный список, не терять)

### Идея пользователя — поиск доп. источников
Пользователь предложил: сделать галочку/опцию "добор знаний в интернете" —
при рерайте система ищет дополнительные источники по теме поста (рус. и
англ. язычные сайты), прикладывает ссылки в пост. Профит:
1. Новость не выглядит "придуманной" — есть подтверждение из независимых
   источников, рост доверия аудитории.
2. SEO/трафик-эффект, если контент дублируется на сайт.
3. Защита от обвинений в дезинформации/фейках.

Реализация: при рерайте параллельно делаем запрос в поисковое API
(Brave Search — есть бесплатный тир, дешевле SerpAPI) по ключевым словам,
извлечённым из оригинального поста. Из топ-5 результатов ИИ сам отбирает
2-3 релевантных (рус. + англ.), добавляет блок "Источники:" в конец
рерайченного поста.

### Идеи, предложенные Claude в ходе обсуждения

**Источники и доверие:**
- Сравнение версий: если русские и английские источники по теме расходятся
  в трактовке — помечать пост "есть разные версии события", это может стать
  фишкой канала (выделяет на фоне обычных репостеров).

**Монетизация и рост:**
- Нативная реклама: каждый N-й пост — рекламный, ИИ органично вписывает
  рекламный блок в контекст поста. Продавать вручную или через биржи типа
  Telega.in.
- SaaS-модель: обернуть систему в простой веб-UI и продавать доступ другим
  владельцам каналов ($20-50/мес). Технически система уже тянет на продукт,
  не просто личный скрипт.
- Авто-дайджест: раз в неделю ИИ сам отбирает топ-5 постов недели по
  просмотрам/реакциям и делает сводный пост-дайджест.

**Качество контента:**
- Стиль-профили: разные промпты под тип контента (новость / мнение /
  инструкция / юмор). ИИ сам классифицирует входящий пост и выбирает голос
  рерайта.
- Авто-обложки: если у поста нет картинки — генерировать через DALL-E/Flux
  (у пользователя уже есть локальный ComfyUI с RX 6600 и опытом AI-генерации
  изображений из других проектов) или брать релевантное фото с Unsplash API.
  Посты с изображением получают значительно больший охват.
- Семантический дубль-чек: не просто хэш текста (который не поймает
  перефразированный дубль из другого канала), а эмбеддинги через тот же
  OpenAI-совместимый API + косинусное сходство. Если похожий пост уже был
  в последние N дней — пропустить или смержить.

**Аналитика и рост:**
- Полная статистика по постам: просмотры, реакции, пересылки на пост —
  через Telegram API (для каналов, где бот — админ, либо через Telethon
  юзер-сессию для get_messages с view_count).
- Умное расписание: анализ истории просмотров → вычисление пиковой
  активности аудитории → автосдвиг времени публикации под пик.
- Growth-трекер: корреляция прироста подписчиков с типом/источником поста —
  видно, что реально растит канал, а что просто шум.

**Защита и безопасность:**
- Антибан-механики: рандомизация задержек между действиями, лимиты в час
  на чтение/публикацию, ротация между несколькими аккаунтами/сессиями.
  Это критично, т.к. Telethon работает через юзер-сессию, а не Bot API —
  риск блокировки юзер-аккаунта при агрессивном парсинге выше, чем у бота.
- Авто-реакция на жалобы: мониторинг негативных реакций на пост, при
  превышении порога — уведомление владельцу, опционально авто-удаление поста.

## Важные технические заметки и риски

1. **Авторские права и контент-политика Telegram.** Рерайт чужого контента
   и репост — это серая зона. Технически система не нарушает ToS Telegram
   напрямую, но юридическая ответственность за контент остаётся на
   владельце канала. Это не блокирующий момент для разработки, но стоит
   держать в голове при масштабировании (особенно для SaaS-модели).

2. **Риск бана юзер-сессии Telethon.** Чтение чужих каналов через юзер-сессию
   не банится Telegram само по себе (это обычное использование клиента),
   но агрессивный паттерн (резкие пики запросов, много каналов сразу) может
   привлечь внимание антиспам-системы. Нужны паузы, рандомизация, разумные
   лимиты — заложено в фазу 4.

3. **Стоимость API-вызовов.** Рерайт + эмбеддинги для дубль-чека + поиск
   источников — три точки расхода токенов/запросов на каждый пост. Нужен
   мониторинг стоимости с первого дня (лог стоимости на пост в БД).

4. **Качество рерайта vs скорость.** Хороший рерайт (не просто синонимайзер)
   требует контекста и инструкций в промпте. Промпт-шаблоны — это то, что
   нужно итерировать дольше всего, не стоит фиксировать один вариант
   навсегда — система должна позволять менять промпты без передеплоя кода
   (хранить в конфиге или БД, не хардкодить в Python).

## Соглашения по разработке для Claude Code

- Все секреты (API-ключи, telegram session string, bot token) — только в
  `.env`, никогда не коммитить, `.env.example` с пустыми значениями в репо.
- Код на Python с type hints, async/await везде где работа с Telethon/aiogram/
  HTTP API (это всё asyncio-нативные библиотеки).
- Миграции БД — через `alembic` с первого дня, даже на SQLite, чтобы переход
  на Postgres в будущем был безболезненным.
- Конфиг — через pydantic `BaseSettings`, не голый `os.environ`.
- Логирование — структурированное (`structlog` или стандартный `logging` с
  JSON-форматтером), с первого дня, не как afterthought.
- Тесты — минимум для бизнес-логики (статус-машина постов, дедупликация,
  парсинг промптов), не обязательно полное покрытие на старте MVP.
