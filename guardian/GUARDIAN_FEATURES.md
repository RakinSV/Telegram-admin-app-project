## P0 — ядро, без этого не запускать

### G01 — Верификация при вступлении (капча)
При появлении нового участника:
1. Immediately restrict (can_send_messages=False).
2. Бот пишет в чат: «@username, добро пожаловать! Ответь на вопрос чтобы
   получить доступ к чату: [ВОПРОС]. У тебя 5 минут.»
3. FSM-состояние: WAITING_CAPTCHA для этого user_id.
4. Правильный ответ → снять restrict, удалить сообщение с капчей,
   отправить приветствие с правилами (удалить через 60 сек).
5. Неправильный → предупреждение, ещё одна попытка (настраиваемо).
6. Тайм-аут (APScheduler) → кик + запись в moderation_log.

Типы капч (настраивается через CAPTCHA_TYPE):
- `math`: «Сколько будет 7 + 4?» — просто, быстро, роботы не всегда
  умеют парсить текст задачи. Варианты ответа кнопками (не текстом —
  боты хуже с кнопками).
- `button`: одна кнопка «Я не робот, нажми сюда» — минимальный порог,
  зато автоматически блокирует ботов без clickjack.
- `question`: тематический вопрос по теме канала (хранится в конфиге,
  несколько вариантов для ротации).

Реализовать все три, переключатель CAPTCHA_TYPE в .env.

### G02 — Приветствие и правила
После успешной верификации:
- Бот пишет приветственное сообщение (шаблон в конфиге/БД, кастомизируемый).
- Кратко: «Запрещено: реклама, ссылки без разрешения, спам. За нарушение —
  предупреждение → мут → бан.»
- Сообщение удаляется автоматически через N секунд (default 120).
- Запись участника в таблицу `members` (join_date, is_verified, warn_count=0).

### G03 — Фильтр стоп-слов (режим 1: keywords)
- Таблица `stop_words` в БД (слово/фраза, добавлено кем, когда).
- При каждом сообщении: нормализация текста (lowercase, убрать дубли пробелов,
  заменить похожие символы — латинские а/е/о на кириллические), проверка
  на вхождение любого стоп-слова.
- Hit → удалить сообщение + выдать варн (через G05) + написать в чат
  «[username] получил предупреждение за нарушение правил».
- Начальный стоп-список: загружается из файла `guardian/data/stopwords_default.txt`
  при первом запуске если таблица пустая.

### G04 — Фильтр ссылок
- Любое сообщение содержащее URL (http/https/t.me/www.) → проверка.
- Если домен не в whitelist (`guardian/data/allowed_domains.txt`) → удалить + варн.
- Telegram-скрытые ссылки (entities типа text_link в объекте Message) тоже
  проверять — это важно, боты часто прячут ссылки за текстом.
- Whitelist по умолчанию пустой (добавлять вручную через /addomain).

### G05 — Система варнов (WarnSystem)
Единый сервис, вызывается из всех фильтров:
```python
await warn_system.add_warn(user_id, chat_id, reason)
# внутри:
# - инкремент warn_count в members
# - запись в moderation_log
# - эскалация по порогам:
#   warn_count == WARN_THRESHOLD_MUTE → restrict на MUTE_DURATION_HOURS
#   warn_count == WARN_THRESHOLD_KICK → kick
#   warn_count >= WARN_THRESHOLD_BAN → ban (перманентный)
# - уведомление в лог-канал
```
TTL-джоба: APScheduler раз в день обнуляет warn_count у участников
у которых последний варн старше WARN_TTL_DAYS.

### G06 — Антифлуд
- Словарь в памяти: {user_id: deque([timestamps])}
- N сообщений за M секунд (N=5, M=10 по умолчанию) → выдать варн.
- Одинаковый текст подряд 2+ раз → удалить дубли + варн.
- Форвардные сообщения из других каналов: настройка ALLOW_FORWARDS (bool).

### G07 — Базовые команды администратора
Все команды — только для пользователей с правами admin/creator в группе.
Проверять через bot.get_chat_member, не хранить список вручную.

```
/warn @user [причина]     — выдать варн вручную
/mute @user [время]       — замутить (1h, 2d, 30m)
/unmute @user             — снять мут
/ban @user [причина]      — забанить
/unban @user              — разбанить
/kick @user               — кикнуть без бана
/check @user              — показать историю (варны, муты, банами)
/addword [слово]          — добавить стоп-слово
/delword [слово]          — удалить стоп-слово
/listwords                — список стоп-слов
/trust @user              — добавить в trusted (исключить из фильтров)
/untrust @user            — убрать из trusted
```
Все действия → запись в moderation_log → уведомление в лог-канал.

### G08 — Лог-канал
При каждом автоматическом действии модерации бот пишет в GUARDIAN_LOG_CHANNEL_ID:
```
🔴 [Удалено] @username | 12:34
Причина: стоп-слово «купить»
Текст: «[первые 200 символов сообщения]»
Варнов: 2/3
[Варн] [Мут] [Бан] — inline-кнопки для быстрого действия
```
При ручных действиях администраторов — тоже логировать (кто сделал, что, кому).

## P1 — следом за MVP

### G09 — AI-фильтр (режим 2)
Классификатор через тот же OpenAI-совместимый клиент что в рерайтере.

Промпт (хранить в `guardian/prompts/spam_classifier.txt`):
```
You are a Telegram chat moderation assistant. Analyze the message and 
respond ONLY with valid JSON, no other text:
{"spam": true|false, "reason": "short reason in Russian", "confidence": 0.0-1.0}

Spam includes: advertising, selling anything, unsolicited links, pyramid schemes,
crypto pumps, job offers without context, hidden advertising.
Not spam: regular questions, opinions, jokes, legitimate discussion.

Message: {message_text}
```

Ответ парсить как JSON. Если `spam=true AND confidence >= 0.75` → действие.
Если `confidence < 0.75` → пропустить (не удалять при неуверенности).
Таймаут API-вызова: 3 секунды. При таймауте — пропустить, логировать.

### G10 — Гибридный режим (режим 3, рекомендуемый по умолчанию)
Логика:
1. Keyword-check (мгновенно, 0 токенов).
   - Явное стоп-слово → действие без AI.
2. Link-check.
   - Ссылка не в whitelist → действие без AI.
3. Эвристики подозрительности (без LLM):
   - Сообщение содержит цены (₽, $, число + руб/usd)?
   - Призывы «напиши в лс», «подробности в профиле»?
   - Невидимые/Zero-Width символы?
   - Форвард из неизвестного источника?
   - Новый аккаунт (< 7 дней) + первое сообщение?
   - Если 2+ признака → передать в AI.
4. AI-check (только для подозрительных ~20% сообщений).
5. Лог: сколько AI-вызовов в день, стоимость, процент спама.

### G11 — Статистика и /stats
```
/stats [день|неделя|месяц]

Модерация за неделю:
• Удалено сообщений: 47
• Выдано варнов: 23
• Мутов: 4, Банов: 2
• Прошли верификацию: 38/41 (93%)
• Стоп-слова: «купить» x12, «заработок» x8
• AI-вызовов: 156, стоимость: ~$0.18
```
Данные из таблицы `moderation_log` + `daily_stats`.
Команда доступна только администраторам.

### G12 — Trusted-пользователи
- Таблица `trusted_users` (user_id, added_by, added_at, reason).
- Trusted полностью обходят spam-filter и link-filter.
- Варны не выдаются.
- Автодоверие: APScheduler проверяет участников у которых
  join_date + AUTO_TRUST_AFTER_DAYS < сегодня И warn_count == 0
  → добавляет в trusted автоматически.
- Репост-бот добавляется в trusted при старте Guardian (через env var
  REPOST_BOT_ID или его username).

### G13 — Конфиг через команды бота (без правки файлов)
```
/setmode keywords|ai|hybrid   — переключить режим фильтрации
/setwarn [мут_порог] [кик_порог] [бан_порог]
/setcaptcha math|button|question
/setmutime [минуты]            — длительность мута
/addomain [домен]              — добавить в whitelist ссылок
/deldomain [домен]
```
Всё в таблице `bot_config` (key, value, updated_by, updated_at).
Загружается при старте, обновляется без перезапуска.

## P2 — продвинутые возможности

### G14 — Антирейд
APScheduler-джоба каждую минуту считает новых участников за последние
RAID_JOIN_WINDOW_MINUTES минут. Если > RAID_JOIN_THRESHOLD:
1. setChatPermissions(all=False) — заморозить все права в группе.
2. Сохранить прежние разрешения (читать текущие перед заморозкой!).
3. Уведомить администратора: «🚨 Рейд-атака! Группа заморожена.
   Вступило N участников за M минут. [Разморозить вручную] [Продолжить авто]».
4. Кнопка авто-разморозки: если 0 новых за RAID_COOLDOWN_MINUTES → восстановить.
5. Логировать всех участников вступивших в период рейда (подозрительные).

### G15 — Анализ профиля нового участника
При вступлении собрать сигналы через Telegram API:
- Нет username → +1 балл подозрительности.
- Нет фото профиля → +1.
- Биография содержит ключевые слова (crypto, заработок, earn) → +2.
- (user.id > ~7 млрд → аккаунт создан после ~2023, не индикатор сам по себе,
  но в сочетании с другим — сигнал).
Score >= N → более строгая капча или автомут до прохождения.
ВАЖНО: не банить только за профиль — это слишком агрессивно.

### G16 — Тихие часы / режимы строгости
Два профиля в `bot_config`:
- `strict`: все фильтры включены, варн за любое нарушение.
- `soft`: стоп-слова работают, но ссылки только логируются (не удаляются),
  первое нарушение — только предупреждение без варна.
Расписание через APScheduler (например strict ночью 22:00-08:00).
Команда /mode strict|soft для ручного переключения.

### G17 — Прирост участников
APScheduler раз в день: count(members WHERE join_date = today) → в daily_stats.
Считать также: сколько прошли верификацию / не прошли (конверсия капчи).
/growth [неделя|месяц] — показать динамику текстовым графиком (спарклайн
через символы ▁▂▃▄▅▆▇█).

## P3 — идеи на будущее

### G18 — Репорты от участников
/report [причина] в ответ на сообщение → уведомление администратору.
Защита: не более 3 репортов в час от одного пользователя.
N репортов на одно сообщение (например 3) → автоудаление + лог.

### G19 — Апелляция бана
Забаненный пишет боту в личку → форма апелляции.
Запрос → администратору с кнопками [Разбанить] [Отклонить].
Хранить историю апелляций.

### G20 — Веб-панель модерации
Простой FastAPI + Jinja2 дашборд для просмотра статистики,
управления стоп-словами, просмотра лога без необходимости писать
команды в бот. Шаг к SaaS-версии Guardian.
