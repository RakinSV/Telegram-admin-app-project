# Бэклог фич — Telegram Content Repost System

> Приоритет: `P0` = без этого не MVP, `P1` = сразу после MVP, `P2` = рост и
> монетизация, `P3` = идеи на будущее, не блокируют ничего.
> Каждая фича — отдельная задача для Claude Code, можно брать по порядку.

## P0 — ядро MVP

### F01 — Управление источниками
Добавление/удаление/просмотр списка отслеживаемых Telegram-каналов.
Хранение в таблице `sources` (id, channel_username, channel_title, is_active,
added_at). Интерфейс — минимум CLI-команды, в идеале — Telegram-бот команды
(`/add_source @channel`, `/list_sources`, `/remove_source @channel`).

### F02 — Telethon listener
Подключение к Telegram через Telethon (юзер-сессия), подписка на новые
сообщения в источниках из `sources`. При получении нового поста — запись в
таблицу `posts` со статусом `new`, сохранение текста, медиа (путь к файлу),
ссылки на оригинал, timestamp.

### F03 — Фильтрация по ключевым словам
Опциональные списки стоп-слов и обязательных слов на канал или глобально.
Применяется сразу при получении поста, до сохранения в очередь (либо
сохраняется со статусом `filtered_out` для прозрачности — лучше второе,
чтобы видеть, что отфильтровалось и почему).

### F04 — Дедупликация (базовая, хэш)
Хэш текста поста (нормализованный — без пробелов/регистра) для отсева точных
дублей между источниками. Простая и быстрая первая линия защиты, до
семантического дубль-чека из P1.

### F05 — Очередь постов (статус-машина)
Таблица `posts` со статусами: `new → filtered_out | duplicate | rewriting →
rewritten → pending_approval → approved | rejected → posted | failed`.
Чёткие переходы, логирование каждой смены статуса с timestamp.

### F06 — Рерайт через OpenAI-совместимое API
Клиент с конфигурируемым `base_url`/`api_key`/`model`. Промпт-шаблон
(хранится в файле/БД, не хардкод) с инструкцией на уникализацию текста
с сохранением смысла. Сохранение результата в `posts.rewritten_text`.
Логирование стоимости запроса (токены) в БД для последующего учёта расходов.

### F07 — Ручное одобрение через Telegram-бота
Бот в личке владельцу присылает рерайченный пост с inline-кнопками
✅ Одобрить / ❌ Отклонить / ✏️ Редактировать. При "Редактировать" — бот
просит прислать новый текст, заменяет `rewritten_text`. Статус меняется
соответственно нажатой кнопке.

### F08 — Публикация в группу
При статусе `approved` — публикация поста (текст + медиа, если есть) в
целевую группу через Telegram Bot API. Обновление статуса на `posted`,
сохранение `message_id` опубликованного поста и timestamp.

### F09 — Конфигурация через .env + pydantic Settings
Все параметры (telegram api_id/api_hash, session string, bot token, target
group id, openai-совместимые ключи) — через `.env`, валидация через pydantic.

### F10 — Базовое логирование и обработка ошибок
Структурированные логи всех этапов пайплайна (получен пост → отфильтрован/
прошёл → рерайчен → отправлен на модерацию → опубликован/отклонён). Retry
с экспоненциальной задержкой для сетевых вызовов (Telegram API, LLM API).

## P1 — сразу после MVP

### F11 — Авто-постинг по расписанию
APScheduler-джоба, которая публикует посты со статусом `approved` (или сразу
`rewritten`, если включён режим без модерации) по расписанию — например,
каждые N часов, либо в заданные временные слоты. Переключается флагом
`auto_post_enabled` в конфиге, не требует пересборки.

### F12 — Несколько целевых групп
Таблица `target_groups` (id, chat_id, title, is_active). Поле в `posts` или
отдельная таблица `post_targets` — какие посты в какие группы идут (может
быть all/subset). UI выбора через бот-команды или конфиг по умолчанию.

### F13 — Семантический дубль-чек (эмбеддинги)
Замена/дополнение хэш-дедупликации (F04) на эмбеддинги через тот же
OpenAI-совместимый API. Косинусное сходство между новым постом и постами
за последние N дней. Порог схожести — конфигурируемый. Хранение векторов —
можно в SQLite как BLOB или через расширение sqlite-vec, без необходимости
поднимать отдельную векторную БД на этом масштабе.

### F14 — Статистика и логи (просмотры, реакции)
Сбор `view_count`/реакций по опубликованным постам через Telegram API (если
бот — админ группы, либо период опроса через Telethon). Таблица
`post_stats` с историей по времени (для построения трендов). Команда бота
`/stats` — сводка за период.

### F15 — Стиль-профили рерайта
Несколько промпт-шаблонов под тип контента (новость/мнение/инструкция/юмор).
Либо ИИ сам классифицирует входящий пост перед рерайтом (отдельный лёгкий
запрос-классификатор), либо привязка стиля к конкретному источнику в
конфиге `sources`.

### F16 — Поиск дополнительных источников (идея пользователя)
Интеграция с Brave Search API. По ключевым словам из оригинального поста —
поиск 5-10 результатов, фильтрация на релевантность (можно через тот же
LLM — "выбери 2-3 наиболее релевантных из списка"), разделение на
русскоязычные и англоязычные. Добавление блока "Источники:" в конец
рерайченного поста со ссылками. Включается опцией на источник или глобально
(галочка "добор знаний").

### F17 — Антибан-механики
Рандомизация задержек между запросами Telethon (не молотить API без пауз).
Лимиты на количество чтений/действий в час. Конфигурируемые джиттер-интервалы.
Это must-have для стабильности, не просто "nice to have" — без этого юзер-
сессия рискует попасть под ограничения.

### F28 — Guardian настраивается по целевым группам (не глобально)
Сейчас Guardian (защита участников: капча, антиспам, антирейд, варны) жёстко
привязан к ОДНОЙ группе через `GUARDIAN_GROUP_ID` в `.env` — при этом
`target_groups` (F12) поддерживает несколько целей публикации. Если целей
несколько, Guardian реально защищает только одну из них. Нужна галочка
"использовать Guardian" на каждой `TargetGroup` в `/targets`, и Guardian
должен уметь обслуживать N групп одновременно (один бот, несколько чатов —
не N ботов). Требует ревизии схемы Guardian (`members`/`warnings`/
`banned_users` и т.д. сейчас неявно на одну группу, нужен `chat_id` как
часть ключа везде) до реализации — см. отдельный анализ перед стартом.
Найдено на аудите ведения групп (2026-07-17), запрошено пользователем явно.

## P2 — рост и монетизация

### F18 — Авто-обложки для постов
Если у поста нет медиа — генерация изображения. Два пути: (а) через
локальный ComfyUI (у пользователя уже настроен ComfyUI + ZLUDA на RX 6600 в
других проектах — можно переиспользовать инфраструктуру) или (б) через
Unsplash API по ключевым словам темы поста (бесплатно, быстрее, но не
уникально). Конфигурируемый выбор стратегии.

### F19 — Умное расписание публикаций
На основе накопленной статистики (F14) — анализ, в какое время аудитория
активнее всего (по просмотрам в первый час после публикации). Автоматический
сдвиг времени публикации под пиковые часы. Нужно minimum 2-3 недели данных
перед тем как это включать осмысленно.

### F20 — Авто-дайджест
Раз в неделю (или другой период) — джоба, которая выбирает топ-N постов по
просмотрам/реакциям за период и собирает их в один сводный пост через LLM.
Публикуется отдельно от обычного потока.

### F21 — Нативная реклама в ленте
Каждый N-й пост (конфигурируемо) — рекламный, ИИ вписывает рекламный текст
органично в формат канала по заданному брифу. Ручное добавление рекламных
брифов через бота, статистика отдельно по рекламным постам.

### F22 — Growth-трекер
Корреляция прироста подписчиков канала (нужен сбор через Telegram API или
сторонние боты-аналитики типа TGStat API) с типом/источником постов.
Отчёт "что реально растит канал".

### F23 — Веб-UI поверх системы (шаг к SaaS)
Простой веб-интерфейс (FastAPI + минимальный фронт, либо Streamlit для
быстрого старта — у пользователя уже есть опыт со Streamlit из проекта
Hedonism Wines) для управления источниками, очередью модерации, статистикой
без необходимости лезть в бота или БД напрямую. Это шаг к продукту, который
можно продавать другим владельцам каналов.

### F29 — Управление уже опубликованным постом
Сейчас редактирование (`/moderation/{id}/edit`) работает только ДО публикации
(пока пост в очереди модерации). После `posted` пост неприкасаем из
админки — нельзя ни поправить текст (Bot API `edit_message_text`/
`edit_message_caption`), ни удалить (`delete_message`), ни закрепить/
открепить (`pin_chat_message`/`unpin_chat_message`). Частый сценарий
владельца канала: заметил опечатку/фактическую ошибку уже после публикации.

### F30 — Модерация комментариев под постами канала
Если канал (не группа) имеет привязанную discussion-группу — Guardian сейчас
работает только с обычными группами, не с этим типом чата. Комментарии под
постами репост-канала никак не фильтруются/не модерируются системой.

### F31 — Полный сбор метрик поста (реакции, форварды)
`PostStat` в БД уже поддерживает `reaction_count`/`forward_count`, но
собирается сейчас только `view_count` (`scheduler/stats.py`). Реакции и
форварды физически доступны через Bot API/Telethon, просто не читаются.

### F32 — Управление инвайт-ссылками и заявками на вступление
Создание/отзыв/лимиты по инвайт-ссылкам целевых групп, одобрение/отклонение
заявок на вступление (join requests, если у группы включено подтверждение
администратором) — сейчас полностью вне системы, только руками в самом
Telegram.

## P3 — идеи на будущее (не блокируют разработку)

### F24 — Сравнение версий источников
Если источники на разных языках расходятся в трактовке темы — пометка поста
"есть разные версии события". Технически сложнее (нужен анализ
противоречий через LLM), отдельная фича после того как F16 стабилен.

### F25 — Авто-реакция на жалобы/негатив
Мониторинг негативных реакций на опубликованный пост, уведомление при
превышении порога, опциональное авто-удаление. Низкий приоритет — нужен
сначала наработанный объём публикаций, чтобы это было релевантно.

### F26 — Ротация между несколькими Telethon-сессиями
Для масштабирования числа источников без риска для одного аккаунта —
балансировка чтения между несколькими юзер-сессиями. Актуально только при
реальном росте числа источников за пределы разумного для одной сессии.

### F27 — SaaS multi-tenant архитектура
Полноценный переход от "система для одного владельца" к multi-tenant
продукту с отдельными аккаунтами, биллингом, изоляцией данных между
клиентами. Это отдельный по объёму проект поверх текущего ядра, не задача
на ближайшие месяцы.

### F33 — Опросы (polls) в ленте
Публикация `send_poll` как отдельный тип поста или довесок к обычному —
вовлечение аудитории без необходимости писать комментарий.

### F34 — Inline-кнопки на опубликованном посте
Ссылка на сайт-первоисточник, "читать полностью", кнопки шаринга — сейчас
посты уходят как голый текст+медиа без разметки клавиатуры.

### F35 — Интеграция с рекламными биржами и учёт дохода
F21 (нативная реклама) даёт только бриф → вписывание в ленту. Нет ни
интеграции с биржами (Telega.in и т.п.), ни учёта фактического дохода от
рекламных постов — этого требует полноценная монетизация, не только
публикация.

### F36 — Приветственные сообщения новым подписчикам канала
Guardian шлёт капчу при вступлении в ЗАЩИЩАЕМУЮ ГРУППУ (F28 — пока одну), но
у КАНАЛА (не группы) вообще нет механизма поприветствовать нового
подписчика — Bot API этого не позволяет напрямую (у ботов нет события "на
канал подписались"), реализуемо только через привязанную группу/бот-чат.

### F37 — Несколько администраторов веб-панели с разными правами
Сейчас один пароль на всю систему (`admin_users` — таблица физически на
одну строку). Актуально, если управлением занимается не один человек.

### F38 — Экспорт/бэкап контента канала
`backup.py` уже бэкапит БД (посты, настройки), но нет экспорта именно
содержимого в независимый от системы формат (например, для передачи новому
владельцу канала или архива для комплаенса).

### F39 — Кросспостинг на другие платформы
Публикация того же (или адаптированного) контента в VK/Instagram и т.п. —
не было в изначальном скоупе проекта, но частая задача владельцев каналов,
растящих присутствие сразу на нескольких площадках.
